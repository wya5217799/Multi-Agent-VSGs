# NE39 Model Routing

> **Repository overlay** — Model-specific routing for the New England 39-bus
> system. Must not be placed in the shared installed `simulink-toolbox` skill.

## Identity

| Field | Value |
|---|---|
| `scenario_id` | `ne39` |
| `model_name` | `NE39bus_v2` |
| Train entry | `scenarios/new_england/train_simulink.py` |
| Config | `scenarios/new_england/config_simulink.py` |
| Model file | `scenarios/new_england/simulink_models/NE39bus_v2.slx` |

## Supported scenario_id values

Only `"ne39"` is valid. Do not use:

- `"new_england"` (registry does not recognise this)
- `"ne_39"`
- `"NE39"`
- any other variant

## Cold-start caveats

- NE39 model (39-bus) first load takes **60–120 s**. This is expected.
- First smoke after a cold start may time out — this is not a model failure.
  Wait for engine to finish loading then retry.
- After cold start, subsequent steps are significantly faster.
- Use async smoke (`harness_train_smoke_start` / `harness_train_smoke_poll`)
  for the first validation on a cold engine to avoid blocking.

## Smoke routing

```
harness_scenario_status    { "scenario_id": "ne39" }  → confirm model_exists + bridge_ready
harness_train_smoke_start  { "scenario_id": "ne39" }  → async smoke (recommended for cold start)
harness_train_smoke_poll   { "task_id": "<id>" }       → poll until done/failed
```

## Known operational notes

- Frequency nominal: **60 Hz**. Do not confuse with Kundur's 50 Hz baseline.
  Using the wrong baseline in plots or reward calculations is a common error.
- Bridge config: `scenarios/new_england/config_simulink.py`.
- Probe scripts for sensitivity testing: `probes/ne39/`.
- Do not run multiple concurrent smoke tasks on the same scenario_id — they
  share MATLAB Engine state.
